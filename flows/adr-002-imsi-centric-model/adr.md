# ADR-002: IMSI-Centric Data Model

**Date**: 2026-03-03  
**Type**: Constraining  
**Status**: DRAFT  
**Author**: Legacy Analysis (/legacy)

---

## Context

The Simbox system manages:
- Multiple GSM modems (dongles)
- Multiple SIM cards (physical and multi-SIM adapters)
- Call routing based on SIM/dongle assignment
- Balance tracking per SIM
- Carrier-specific operations

The system needs a consistent identifier to:
- Track SIM cards across different dongles
- Associate state, settings, and statistics
- Link KI credentials to physical SIMs
- Route calls to appropriate SIM/dongle

## Problem

What should be the primary identifier for SIM cards in the system?

Options considered:
- **ICCID**: Physical card identifier, unique but tied to hardware
- **Phone Number**: User-facing, but can change (number portability)
- **IMSI**: Network-level subscriber identity, stable and unique
- **Dongle ID**: Tied to hardware, but SIMs move between dongles

## Decision

**Use IMSI (International Mobile Subscriber Identity) as the primary key** for all SIM-related data.

### IMSI Format

- 15 digits (sometimes stored as 18 with leading zeros)
- Structure: MCC (3) + MNC (2-3) + MSIN (9-10)
- Example: `250201234567890` (Russia, MegaFon)

### File Naming Convention

All SIM-related files use IMSI as filename:

```
/var/simbox/sim/statistics/250201234567890.balance    # Balance
/var/simbox/sim/statistics/250201234567890.op_t       # Operator type
/var/simbox/sim/settings/250201234567890.tarif        # Tariff name
/var/simbox/sim/settings/250201234567890.number       # Phone number
/var/simbox/sim/settings/250201234567890.owner        # Owner ID
/var/simbox/sim/settings/250201234567890.group        # Routing group
/var/svistok/sim/state/250201234567890.dongle         # Current dongle
```

### Cross-Reference Files

```
/var/svistok/dongles/state/dongle01.imsi    # Current IMSI in dongle
/var/svistok/sim/state/250201234567890.dongle  # Current dongle for SIM
```

### Code Examples

```php
// modules/sim.php - Balance operations by IMSI
function save_balance($imsi, $balance) {
    $file = "/var/simbox/sim/statistics/{$imsi}.balance";
    file_put_contents($file, $balance);
}

function upload_balance($imsi, $balance) {
    // Sync to central server
    $url = "http://simserver:8122/balance.php?imsi={$imsi}&balance={$balance}";
    file_get_contents($url);
}
```

## Consequences

### Positive

- **Stability**: IMSI doesn't change (unlike phone number)
- **Network Integration**: IMSI used in GSM protocol, matches Asterisk logs
- **Uniqueness**: Globally unique identifier
- **Carrier Info**: MCC/MNC prefix reveals carrier/region
- **Consistency**: Single identifier across all subsystems

### Negative

- **Opacity**: IMSI not user-friendly (vs phone number)
- **Multi-SIM Complexity**: Multi-SIM adapters can change IMSI
- **Privacy**: IMSI is sensitive subscriber data
- **Hardcoded**: Some parsers assume IMSI length (15 vs 18 digits)

### Security Implications

- IMSI exposure allows subscriber tracking
- File permissions critical (world-readable IMSI files = privacy leak)
- Central server transmission should be encrypted (currently HTTP)

## Implementation Notes

- IMSI stored as string (not integer) to preserve leading zeros
- Some files use 18-digit format (padded), others 15-digit
- Parser regex should handle both formats
- Multi-SIM adapters: IMSI changes when profile switched
