# ADR-003: Multi-SIM Adapter Support

**Date**: 2026-03-03  
**Type**: Enabling  
**Status**: DRAFT  
**Author**: Legacy Analysis (/legacy)

---

## Context

The Simbox system operates multiple GSM modems for call routing. Traditional SIM cards are:
- Physical hardware tied to one IMSI/ICCID
- Expensive to replace when blocked by carriers
- Limited in number per modem

Multi-SIM adapters (programmable SIM cards) provide:
- Multiple SIM profiles on one physical card
- Remote reprogramming via AT commands
- KI (cryptographic key) rotation capability
- IMEI changing support

## Problem

How to support programmable multi-SIM adapters that can:
- Store multiple SIM profiles (IMSI/KI/ICCID)
- Be reprogrammed remotely via software
- Rotate credentials for anti-detection
- Work with existing Asterisk chan_dongle infrastructure

## Decision

**Support two types of multi-SIM adapters** with different programming interfaces:

### Adapter Types

| Type | Name | Programming Interface |
|------|------|----------------------|
| Type 9 | STK Adapter | SIM Toolkit AT commands |
| Type 16 | SuperSIM | CSIM APDU commands |

### Type 9: STK Adapter

Uses GSM 11.14 SIM Toolkit commands:
```
AT+STGR=<tag>    // Send terminal response
AT+STIN=<tag>    // Check available commands
AT+STSF=...      // SIM toolkit setup
```

Profile switching via STK menu navigation.

### Type 16: SuperSIM

Uses 3GPP CSIM (CDMA SIM) APDU commands:
```
AT+CSIM=<len>,"<apdu>"    // Send raw APDU to SIM
```

Example APDU for profile switch:
```
AT+CSIM=14,"A0A4000002A000"    // Select DF
AT+CSIM=10,"00A4000402"        // Select EF
```

### KI Programming Flow

```
1. Fetch new KI from central server (new_ki.php)
2. Put adapter in programming mode
3. Send KI programming APDU sequence
4. Verify programming success
5. Update local state files
6. Upload credentials to central server
```

### File: `modules/ki.php`

Key functions:
```php
function program_ki_type9($dongle, $ki, $imsi, $iccid) {
    // STK-based programming for Type 9
    // Uses AT+STGR, AT+STIN commands
}

function program_ki_type16($dongle, $ki, $imsi, $iccid) {
    // CSIM APDU-based programming for Type 16
    // Uses AT+CSIM commands
}
```

### Central Server Integration

```php
// Fetch new KI credentials
$ki_data = file_get_contents("http://simserver:8122/new_ki.php");
// Response: KI, IMSI, ICCID, SMSC

// Upload programmed credentials
file_get_contents("http://simserver:8122/upload_ki.php?imsi=$imsi&...");
```

## Consequences

### Positive

- **Flexibility**: Switch SIM profiles without physical access
- **Cost Reduction**: One adapter replaces multiple physical SIMs
- **Anti-Detection**: Rotate KI/IMSI to evade carrier blocking
- **Central Management**: KI credentials managed centrally
- **Automation**: KI rotation can be automated

### Negative

- **Complexity**: Two different programming interfaces
- **Reliability**: APDU commands can fail, leaving adapter in unknown state
- **Security**: KI transmitted over HTTP (unencrypted)
- **Carrier Detection**: Pattern detection still possible despite rotation
- **Hardware Dependency**: Specific adapter models required

### Security/Ethical Concerns

- Multi-SIM adapters commonly used for SIM-box fraud
- KI rotation enables evasion of carrier fraud detection
- System designed for high-volume GSM termination
- Potential legal implications in many jurisdictions

## Implementation Notes

- Type 9: Simpler, but limited functionality
- Type 16: Full SIM programming, more complex
- KI length: 32 hex characters (128-bit key)
- IMSI length: 15-18 digits
- ICCID length: 19-20 digits
