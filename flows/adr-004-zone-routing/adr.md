# ADR-004: Zone-Based Call Routing

**Date**: 2026-03-03  
**Type**: Enabling  
**Status**: DRAFT  
**Author**: Legacy Analysis (/legacy)

---

## Context

The Simbox system routes calls through multiple GSM carriers across different regions:
- St. Petersburg (Tele2, Beeline, MegaFon, MTS, Rostelekom)
- Moscow (MegaFon)
- Ukraine (Kievstar)
- Belarus (Velcom, Life)

Each carrier/region combination has:
- Different dialing rules (local, federal, international)
- Different tariff structures
- Different routing preferences
- Different SIM groups (work, reserve, low balance, blocked)

## Problem

How to route outbound calls through the appropriate SIM/dongle based on:
- Destination number format (local, mobile, international)
- Carrier preferences (cost optimization)
- SIM availability and balance
- Special routing modes (VIP, test, navigation)

## Decision

**Implement zone-based routing** using Asterisk dialplan configuration files organized by carrier and region.

### Zone File Structure

```
asterisk/extensions/
├── extensions_dial.conf         # Main dial logic
├── extensions_zones.conf        # Zone includes
└── zones/
    ├── extensions_all_spb.conf      # All carriers SPb
    ├── extensions_bee_spb.conf      # Beeline SPb
    ├── extensions_meg_spb.conf      # MegaFon SPb
    ├── extensions_mts_spb.conf      # MTS SPb
    ├── extensions_tele2_spb.conf    # Tele2 SPb
    ├── extensions_rostel_spb.conf   # Rostelekom SPb
    ├── extensions_kievstar_ua.conf  # Kievstar Ukraine
    ├── extensions_velcom_by.conf    # Velcom Belarus
    └── extensions_life_by.conf      # Life Belarus
```

### Dialplan Pattern Matching

Each zone file contains pattern-matched extensions:

```asterisk
; extensions_tele2_spb.conf
[extensions-tele2-spb]
; Local SPb numbers (812)
exten => _812XXXXXXX,!
    Set(DONGLE_GROUP=tele2_spb_work)
    Set(ROUTE_COST=1)
    Goto(dial-via-gsm,1)

; Federal Russian numbers (8 XXXXXX)
exten => _8[2-9]XXXXXXXX,!
    Set(DONGLE_GROUP=tele2_federal)
    Set(ROUTE_COST=2)
    Goto(dial-via-gsm,1)

; International numbers (+XX...)
exten => _+X.!
    Set(DONGLE_GROUP=tele2_intl)
    Set(ROUTE_COST=3)
    Goto(dial-via-gsm,1)
```

### SIM Group Assignment

SIMs assigned to groups based on state:

```
GROUP_WORK_PRE/OK    # Active working SIMs
GROUP_ZAPAS_PRE/OK   # Reserve SIMs
GROUP_LOW_BALANCE    # Low balance, limited routing
GROUP_BLOCKED        # Blocked, no routing
```

### Special Routing Modes

Encoded in dialed number suffix:

| Suffix | Mode | Behavior |
|--------|------|----------|
| `#SOU` | Sound | Audio test mode |
| `#PRE` | Pre-answer | Pre-answer routing |
| `#POS` | Post-answer | Post-answer routing |
| `#MAY` | May | Special routing |
| `#MAG` | Mag | Special routing |
| `#VIP` | VIP | Priority routing |
| `#NAV` | Nav | Navigation mode |

### Call Flow

```
1. User dials number (e.g., +79991234567#VIP)
2. Asterisk extracts special mode (#VIP)
3. Dialplan matches destination pattern
4. Zone file determines carrier group
5. SIM selected from group (round-robin or least-cost)
6. Call routed via chan_dongle to GSM
7. Central server notified (in.php)
```

## Consequences

### Positive

- **Flexibility**: Easy to add new carriers/regions
- **Cost Optimization**: Route through cheapest carrier
- **Load Balancing**: Distribute calls across SIMs
- **Failover**: Automatic fallback to reserve SIMs
- **Special Handling**: VIP, test modes supported

### Negative

- **Complexity**: Many zone files to maintain
- **Configuration Drift**: Zones can become inconsistent
- **Testing Burden**: Each zone needs testing
- **Hardcoded Logic**: Routing rules in dialplan (not database)

### Operational Concerns

- Zone changes require Asterisk reload
- No real-time routing based on current balance
- SIM group assignment updated by cron, not event-driven

## Implementation Notes

- Zone files included from main dialplan
- Pattern matching uses Asterisk regex syntax
- DONGLE_GROUP variable controls SIM selection
- Central server notified for all calls
