# ADR-001: File-Based State Storage

**Date**: 2026-03-03  
**Type**: Constraining  
**Status**: DRAFT  
**Author**: Legacy Analysis (/legacy)

---

## Context

The Simbox system requires persistent storage for:
- Dongle states (IMSI, ICCID, connection status)
- SIM card states (balance, tariff, phone number, limits)
- System configuration (paths, settings)
- Call statistics and logs
- KI credentials for multi-SIM adapters

The system operates in a telecom environment with:
- Multiple concurrent processes (cron jobs, Asterisk, web interface)
- Need for high reliability and crash recovery
- Shell scripts and PHP sharing state
- Real-time state updates from USSD/SMS responses

## Problem

How should the system store and manage state data across multiple components (Shell, PHP, Asterisk) while ensuring:
- Concurrent access from multiple processes
- Crash recovery without data corruption
- Simple debugging and manual intervention
- No external database dependencies

## Decision

**Use flat files for all state storage** instead of a database.

### Directory Structure

Two parallel directory trees:

```
/var/svistok/           # Runtime state (dongle-centric)
├── dongles/
│   └── state/          # Dongle states (.imsi, .iccid, .str_state)
├── sim/
│   ├── state/          # SIM state files (.dongle mapping)
│   ├── settings/       # SIM configuration
│   ├── statistics/     # Balance, call stats
│   └── limits/         # SIM limits
└── lists/              # Lists (dongles.list, imsi.list)

/var/simbox/            # Application data (SIM-centric)
├── sim/
│   ├── state/
│   ├── settings/
│   └── statistics/
└── plan/               # Calling plans
```

### File Naming Convention

- One file per entity (SIM/dongle)
- IMSI as primary filename identifier
- Simple key-value or raw value content
- Atomic writes via temp files when needed

### Access Pattern

```php
// PHP utility function (head.php)
function file_get_contents_def($file, $default = '') {
    if (file_exists($file)) {
        return trim(file_get_contents($file));
    }
    return $default;
}

function exec_s($cmd) {
    // Execute with sudo for system commands
    return shell_exec("sudo " . escapeshellcmd($cmd));
}
```

## Consequences

### Positive

- **Simplicity**: No database setup, schema migrations, or ORM needed
- **Debugging**: State can be inspected with `cat` command
- **Crash Recovery**: File system journaling provides consistency
- **Portability**: Easy to backup/restore with `tar`
- **Shell Integration**: Native shell script access without DB drivers
- **Concurrency**: File locking sufficient for this scale

### Negative

- **Query Limitations**: No SQL queries, filtering requires iteration
- **Scalability**: Performance degrades with many files per directory
- **Atomicity**: No transactions, multi-file updates not atomic
- **Validation**: No schema enforcement, data types not validated
- **Security**: File permissions critical, no access control beyond OS

### Technical Debt

- Dual directory structure (`svistok` vs `simbox`) creates confusion
- No centralized validation of file content
- Race conditions possible without proper locking
- Hard to query across multiple state files

## Implementation Notes

- Utility functions in `head.php` provide consistent file access
- Shell scripts use `config.sh` for path definitions
- State files are typically small (<1KB each)
- Directory permissions: owned by asterisk/www-data users
